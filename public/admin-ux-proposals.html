Admin UX: Identität, Rolle und Tenant-Kontext

Problem: Im aktuellen Admin ist unklar, ob man als eingeladener Workspace-User, Tenant-Admin oder globaler Betreiber unterwegs ist. Diese drei Vorschläge lösen das über unterschiedliche Informationsarchitekturen.

Empfehlung Variante B ist wahrscheinlich die beste SaaS-MVP-Basis: links Scope/Workspace, oben Identität, rechts Rechte. Sie bleibt verständlich, auch wenn später Billing, Rollen und mehrere Tenants dazukommen.

A: Kontextleiste über bestehendem Tenant Admin

Minimal-invasiv. Wir behalten die aktuelle Tenant-Admin-Struktur, ergänzen aber oben eine eindeutige Scope-Leiste: Wer bin ich, in welchem Workspace bin ich, welche Rolle habe ich, und was darf ich hier?

Gut für: schnelle Korrektur + Wenig Umbau, klare Orientierung sofort - Super Admin und Tenant Admin bleiben getrennte Seiten

B: Einheitliche Admin Shell mit Workspace Switcher

Ein klares SaaS-Admin-Gerüst: Links alle Tenants/Workspaces, Mitte die aktive Arbeitsfläche, rechts Kontext zu Rolle und Rechten. Super-Admin-Funktionen erscheinen nur, wenn der User sie wirklich hat.

Gut für: SaaS-MVP + Beste Orientierung und skalierbar für mehrere Workspaces - Braucht etwas mehr Frontend-Umbau als Variante A

C: Super Admin als eigene Betreiber-Konsole

Radikale Trennung: Normale Tenant-User sehen nie globale Navigation. Super Admins landen zuerst in einer dunklen Betreiber-Konsole und springen von dort gezielt in einen Tenant als Owner/Admin.

Gut für: klare Sicherheitsgrenzen + Super Admin ist unmissverständlich - Zwei stärkere Produktbereiche statt einer gemeinsamen Shell
